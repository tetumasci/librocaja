# Prompt para Claude Code — Libro de Caja, nuevas features

## Contexto del proyecto
PWA de finanzas personales (HTML/CSS/JS vanilla, sin frameworks, sin build
step). Archivos: `index.html`, `styles.css`, `app.js`, `manifest.json`,
`sw.js`. Persistencia 100% local con `localStorage`, clave
`libro-caja-data-v1`. Identidad visual "libro de caja": papel crema
(`--paper: #FAF6EF`), tinta (`--ink: #1F1B16`), terracota para gastos
(`--expense: #B8512F`), oliva para ingresos (`--income: #4A5D3A`), dorado
de acento (`--accent: #C99A3D`). Tipografía: Source Serif 4 para display,
Inter para UI, JetBrains Mono para números. Mantené esta identidad en todo
lo nuevo — no introduzcas paletas o tipografías distintas.

Antes de tocar código, leé `index.html`, `styles.css` y `app.js` completos
para entender la estructura actual (vistas tipo overlay full-screen,
modales tipo bottom-sheet, patrón de `state` global + `saveState()` /
`loadState()` + funciones `renderX()` por sección).

Implementar las features en el orden listado abajo, cada una como un paso
separado y probado antes de pasar a la siguiente. No mezclar varias
features en un mismo cambio grande.

## PARTE 1 — Día a día (libro de caja)

### 1. Múltiples cuentas/billeteras
- Nueva entidad `accounts`: `{id, name, icon, type}` donde `type` es
  `'cash' | 'bank' | 'digital'` (efectivo, banco, billetera virtual tipo
  Mercado Pago). Cuentas por defecto sugeridas: "Efectivo", "Banco",
  "Mercado Pago" — pero editable/ampliable igual que las categorías.
- Cada movimiento (`entries`) suma un campo `accountId`.
- En el modal de carga de movimiento, agregar selector de cuenta (mismo
  patrón visual que el selector de categorías).
- En el resumen mensual, agregar la posibilidad de ver el saldo
  desglosado por cuenta, no solo el total.
- Migración: los movimientos existentes sin `accountId` deben asignarse a
  una cuenta "General" por defecto al cargar `loadState()`, para no romper
  datos previos.

### 2. Gastos fijos / recurrentes
- Nueva entidad `recurringExpenses`: `{id, name, amount, categoryId,
  accountId, dayOfMonth, active}`.
- Pantalla o sección (dentro de Ajustes está bien) para crear/editar/
  pausar gastos recurrentes.
- Al abrir la app, si estamos en un mes nuevo y un gasto recurrente activo
  todavía no tiene un `entry` generado para ese mes, generarlo
  automáticamente como entry normal (mismo tipo de dato que un gasto
  cargado a mano, con un flag `autoGenerated: true` para distinguirlo
  visualmente con un pequeño ícono en el listado).
- Pedir confirmación la primera vez que se auto-genera un mes de gastos
  fijos (toast o modal simple), no hacerlo en silencio.

### 3. Presupuestos por categoría
- Nueva entidad `budgets`: `{categoryId, monthlyLimit}`.
- Configurable desde Ajustes, junto al gestor de categorías existente.
- En la vista de Reportes, cada barra de categoría debe mostrar el
  porcentaje consumido del presupuesto si existe uno definido (ej. barra
  con marca o color distinto al pasar el 80% y el 100% del límite).
- Sugerencia dinámica nueva en el sistema de `computeSuggestion()` ya
  existente: avisar cuando una categoría con presupuesto definido supera
  el 90% del límite mensual.

### 4. Vista calendario de racha (estilo GitHub contribution graph)
- Nueva vista accesible desde algún lado del Libro principal (ej. tocando
  el número de racha en `streak-bar`, o un ícono nuevo).
- Grilla de cuadraditos, uno por día, de los últimos ~3-4 meses. Color más
  intenso = más movimientos cargados ese día, vacío/gris = sin carga.
  Reusar la paleta existente (tonos de `--accent` o `--income` para la
  intensidad, no inventar colores nuevos).
- Mantenerlo simple: no hace falta tooltip con el detalle de cada día en
  la v1, alcanza con el patrón visual.

### 5. Comparativa con inflación (Argentina)
- Mostrar en Reportes el gasto del mes ajustado por inflación vs el mes
  anterior, para que el usuario vea si gastó más "en términos reales" o
  solo por el efecto inflacionario.
- Como esta app no tiene backend, el dato de inflación mensual del INDEC
  debe ser un valor que el usuario carga/actualiza manualmente desde
  Ajustes (input simple: "inflación de este mes: __%"), guardado en una
  nueva entidad `inflationRates: {'2026-06': 4.2, ...}`. No intentar
  conectar a ninguna API externa — la app es offline-first.
- Cálculo: gasto real mes actual vs gasto mes anterior ajustado por el
  índice cargado. Mostrar como métrica adicional en la grilla de métricas
  ya existente (`metric-grid`).

## PARTE 2 — Nuevo módulo: "Plan" (inversiones y ahorro a largo plazo)

Esta es una sección nueva, separada del libro de caja diario — agregar
como una quinta solapa en el bottom nav (`nav-item` con ícono de
tendencia/gráfico ascendente) que abre un nuevo `view-overlay` llamado
`view-plan`, siguiendo el mismo patrón que `view-stats` / `view-goals` /
`view-settings`.

### Modelo de datos
Nueva entidad `investmentPlans`: array de objetos:
```
{
  id,
  name,                 // ej "Swiss Medical 25 años"
  type,                 // 'compound_interest' | 'manual' (para futuro,
                         // hoy solo se implementa compound_interest)
  currency: 'USD',
  monthlyContributionUSD,  // ej 100
  annualRatePct,         // ej 6.5
  startDate,             // ISO de inicio
  termYears,             // ej 25
  contributions: [
    { date, amountUSD, amountARSPaid, exchangeRateUsed }
  ]
}
```

El usuario carga `amountARSPaid` cada vez que paga la cuota (porque a él
se la cobran en pesos al tipo de cambio del día), pero el cálculo de
interés compuesto y la proyección viven enteramente en USD —
`amountARSPaid` es un dato informativo de cuánto le costó en pesos ese
mes puntual, no afecta la proyección.

### Cálculo de interés compuesto
Interés compuesto mensual sobre aportes mensuales fijos (anualidad):
- Tasa mensual = `(1 + annualRatePct/100)^(1/12) - 1`
- Valor futuro proyectado a `termYears`: fórmula de valor futuro de una
  anualidad ordinaria con aportes mensuales constantes.
- Mostrar también el valor acumulado real a la fecha (sumando aportes +
  interés generado mes a mes desde `startDate` hasta hoy), no solo la
  proyección a 25 años.

### UI de la vista "Plan"
- Tarjeta por cada plan cargado mostrando: nombre, aportado hasta la
  fecha (USD), valor proyectado a término (USD), un gráfico simple de
  curva de crecimiento (puede ser barras o línea simple en CSS/canvas,
  mismo nivel de simplicidad que el `trend-chart` que ya existe en
  Reportes — no hace falta una librería de gráficos).
- Botón "+ nuevo plan" para cargar otro plan a futuro (el usuario
  mencionó que también invierte en Cocos — dejar el modelo flexible para
  que en el futuro pueda cargar un segundo plan tipo `type: 'manual'`
  donde simplemente registra aportes y valuaciones sin fórmula de interés
  compuesto fija, pero esa parte NO hace falta implementarla todavía,
  solo no cerrar la puerta en el modelo de datos).
- Botón "registrar pago de este mes" en cada plan, que abre un modal
  simple pidiendo el monto en pesos pagado (y opcionalmente el tipo de
  cambio usado, si no se carga se puede dejar vacío) y agrega una entrada
  a `contributions` con el aporte fijo en USD correspondiente.

### Qué NO hacer en esta parte
- No conectar a ninguna cotización de dólar en tiempo real — todo carga
  manual.
- No mezclar el módulo Plan con el saldo/balance del libro de caja
  diario — son sistemas separados a propósito (uno es flujo de caja
  mensual, el otro es ahorro de largo plazo). No sumar el valor del plan
  al balance mensual en ningún lado.

## Orden de implementación sugerido
1. Múltiples cuentas (es la base que tocan varias features después)
2. Gastos recurrentes
3. Presupuestos por categoría
4. Calendario de racha
5. Inflación
6. Módulo Plan completo (es la pieza más grande, dejarla para el final)

## Qué probar al terminar cada feature
- Que `loadState()` no rompa con datos guardados de versiones anteriores
  de la app (campos nuevos deben tener default sensato si no existen).
- Que el flujo de apertura/cierre de cualquier vista o modal nuevo respete
  la corrección de overlays ya hecha (un solo overlay visible a la vez).
- Que `formatMoney()` se use consistentemente para todo lo que se muestre
  en pesos, y se use un formateador separado y claramente etiquetado
  "USD" para todo lo que se muestre en dólares (no mezclar símbolos $ sin
  aclarar la moneda en ningún lado de la UI nueva).
